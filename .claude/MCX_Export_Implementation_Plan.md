# MCX Export Implementation Plan

## Goal

Export the voxelized head volume and optode positions from NIRWizard into MCX-compatible files, enabling GPU-accelerated photon transport simulation for sensitivity profiling and Jacobian computation.

## Context: What Exists Today

Rust backend has:
- `SubjectAnatomy` with skull/csf/grey_matter/white_matter as `SceneObject` meshes
- `head_labels.mgz` generated by `run_voxelization()` with labels 0-4
- `OptodeLayout` with sources/detectors positioned in 3D scene space
- Scene transforms for both anatomy and optode layout
- `neuroformats` crate for reading MGH files
- Full dependency set: nalgebra, serde_json, parry3d, kiddo

What is missing:
- The `vox2ras` matrix is not stored after anatomy loading
- No MCX domain types
- No coordinate transform from scene space to voxel space
- No export command
- No way to invoke MCX
- No fluence loading or Jacobian computation

## MCX Input Requirements

MCX needs two files per simulation run:

1. A binary volume file: flat array of `u8` values, one byte per voxel, where each byte is a tissue label index (0 = background/air, 1+ = tissue types). Row-major order (C layout).

2. A JSON input file containing:
   - Session settings (photon count, output type)
   - Source position and direction in voxel grid coordinates
   - Detector positions in voxel grid coordinates
   - Domain dimensions and volume file reference
   - Media array: optical properties [mua, mus, g, n] per tissue label

## Step 1: Store Voxel Metadata in SubjectAnatomy

Currently `SubjectAnatomy` only stores the mesh `SceneObject`s. We need to also store the voxel grid metadata so the MCX exporter can access it later without re-reading files.

### 1.1 Extend SubjectAnatomy

File: `src-tauri/src/domain/anatomy.rs`

Add three fields:

```rust
pub struct SubjectAnatomy {
    pub skull: Option<SceneObject>,
    pub csf: Option<SceneObject>,
    pub grey_matter: Option<SceneObject>,
    pub white_matter: Option<SceneObject>,

    // Voxel grid metadata for MCX export
    pub labels_path: Option<PathBuf>,
    pub vox2ras: Option<na::Matrix4<f64>>,
    pub volume_dim: Option<[usize; 3]>,
}
```

- `labels_path`: absolute path to the `head_labels.mgz` file so we can re-read the raw bytes at export time without keeping the entire 256x256x256 array in memory
- `vox2ras`: the 4x4 affine matrix from the MGH header that maps voxel indices to RAS (scanner) coordinates. This is the critical coordinate transform.
- `volume_dim`: the [nx, ny, nz] dimensions of the voxel grid

### 1.2 Extract vox2ras from MGH Header

File: `src-tauri/src/io/anatomy_importer.rs`

The `neuroformats::FsMgh` header contains the fields needed to reconstruct the vox2ras matrix. The standard FreeSurfer formula is:

```
vox2ras = [ Mdc * diag(delta)  |  P0 ]
          [     0   0   0      |   1  ]
```

Where:
- `Mdc` is the 3x3 direction cosine matrix (columns are x_r/x_a/x_s, y_r/y_a/y_s, z_r/z_a/z_s)
- `delta` is the voxel size vector [xsize, ysize, zsize]
- `P0` is the RAS coordinate of voxel (0,0,0)

Write a helper function:

```rust
fn extract_vox2ras(header: &neuroformats::FsMghHeader) -> na::Matrix4<f64> {
    // Build from header fields: x_r, x_a, x_s, y_r, y_a, y_s, z_r, z_a, z_s
    // delta (voxel sizes), and c_r, c_a, c_s (center RAS)
    // P0 = center_ras - Mdc * diag(delta) * (dim / 2)
    //
    // Consult neuroformats source to confirm exact field names
}
```

Important: verify the field names by inspecting the `neuroformats` crate source. The header struct likely has fields like `x_r`, `x_a`, `x_s` (or similar) for the direction cosines, and `xsize`, `ysize`, `zsize` for voxel dimensions.

### 1.3 Populate Fields During Anatomy Loading

In `load_subject_anatomy()`, after `run_voxelization()` completes, read the header from the generated `head_labels.mgz` (or reuse the `aseg_mgh.header` which shares the same grid):

## Step 3: Implement Coordinate Transforms

File: `src-tauri/src/domain/mcx.rs` (or a new `src-tauri/src/domain/transforms.rs`)

This is the most critical and error-prone step. The chain is:

```
Optode position (SNIRF raw, mm)
    |
    v  apply OptodeLayout.transform (user positioned the probe)
Scene world position (mm, RAS-aligned)
    |
    v  apply ras2vox (inverse of vox2ras from MGH header)
Voxel grid coordinates (float indices into NxNyNz grid)
    |
    v  MCX uses these directly (supports float positions)
```

### 3.1 world_to_voxel Function

```rust
pub fn world_to_voxel(
    local_pos: &na::Vector3<f64>,      // optode position in layout-local space
    layout_transform: &Transform,       // the probe's scene transform
    vox2ras: &na::Matrix4<f64>,         // from MGH header
) -> Result<na::Vector3<f64>, String> {
    // Step 1: local -> world
    let world = layout_transform.matrix()
        .transform_point(&na::Point3::from(*local_pos));

    // Step 2: world (RAS) -> voxel
    let ras2vox = vox2ras.try_inverse()
        .ok_or("vox2ras matrix is not invertible")?;
    let voxel = ras2vox.transform_point(&world);

    Ok(na::Vector3::new(voxel.x, voxel.y, voxel.z))
}
```

### 3.2 Compute Source Direction

The source direction should point inward from the scalp surface toward the brain. Two options:

Option A (simple): direction from optode toward the volume center.

```rust
let center = na::Vector3::new(nx as f64 / 2.0, ny as f64 / 2.0, nz as f64 / 2.0);
let dir = (center - source_voxel).normalize();
```

Option B (better): use the scalp surface normal at the optode position. This requires raycasting from the optode position onto the skull mesh and using the hit normal. You already have parry3d and the skull SceneObject — this is a future improvement.

Start with Option A.

### 3.3 Bounds Checking

After converting to voxel coordinates, verify:
- All coordinates are within [0, dim) for each axis
- The voxel at the source position is label 0 (air) or label 1 (skull surface)
- If the source is inside tissue, warn the user that probe alignment may be wrong

### 3.4 Test

Create a test with known geometry:
- A 60x60x60 volume with 1mm isotropic voxels
- An identity vox2ras (so world coords = voxel coords)
- Place a source at world position [30, 30, 0]
- Verify the voxel coordinate is [30, 30, 0]
- Then test with a non-trivial vox2ras (translated, scaled) and verify the transform is correct

## Step 4: Implement the Export Command

File: `src-tauri/src/commands/mcx.rs`

### 4.1 Export Volume Binary File

Read `head_labels.mgz` from the stored path, extract the u8 label array, and write it as a flat binary file in row-major (C) order:

```rust
fn write_volume_binary(
    labels_path: &Path,
    output_path: &Path,
) -> Result<[usize; 3], String> {
    let mgh = FsMgh::from_file(labels_path)?;
    let data = mgh_as_u8(&mgh)?;
    let (nx, ny, nz, _) = data.dim();

    // Flatten to row-major bytes
    let flat: Vec<u8> = data.iter().cloned().collect();
    fs::write(output_path, &flat)?;

    Ok([nx, ny, nz])
}
```

Note: verify that ndarray's default iteration order matches what MCX expects. MCX reads volumes in column-major (Fortran/MATLAB) order by default (`-a 0`). If ndarray iterates in row-major order, you need to either transpose the array before writing, or pass `-a 1` to MCX. This is a common source of bugs.

### 4.2 Build and Write MCX JSON

Create the `McxInput` struct from session data:

```rust
#[tauri::command]
pub fn export_mcx_input(
    output_dir: String,
    source_index: usize,
    photon_count: Option<u64>,
    state: tauri::State<AppState>,
) -> Result<McxExportResult, String> {
    let session = state.session.read().map_err(|e| e.to_string())?;

    let anatomy = session.subject_anatomy.as_ref()
        .ok_or("No anatomy loaded. Load an MRI first.")?;
    let layout = session.optode_layout.as_ref()
        .ok_or("No optode layout. Load a SNIRF file first.")?;
    let vox2ras = anatomy.vox2ras.as_ref()
        .ok_or("No voxel coordinate transform available.")?;
    let labels_path = anatomy.labels_path.as_ref()
        .ok_or("No voxel labels file available.")?;

    // Validate source index
    if source_index >= layout.sources.len() {
        return Err(format!("Source index {} out of range (have {} sources)",
            source_index, layout.sources.len()));
    }

    // Create output directory
    let out_dir = Path::new(&output_dir);
    fs::create_dir_all(out_dir)?;

    // Write volume binary
    let vol_filename = "volume.bin";
    let dims = write_volume_binary(labels_path, &out_dir.join(vol_filename))?;

    // Convert source position
    let source = &layout.sources[source_index];
    let source_voxel = world_to_voxel(
        &source.position, &layout.transform, vox2ras
    )?;
    let center = na::Vector3::new(
        dims[0] as f64 / 2.0,
        dims[1] as f64 / 2.0,
        dims[2] as f64 / 2.0,
    );
    let dir = (center - source_voxel).normalize();

    // Convert all detector positions
    let detectors: Vec<McxDetector> = layout.detectors.iter()
        .map(|det| {
            let pos = world_to_voxel(&det.position, &layout.transform, vox2ras)
                .unwrap_or(na::Vector3::zeros());
            McxDetector {
                pos: [pos.x, pos.y, pos.z],
                radius: 2.0,
            }
        })
        .collect();

    // Build MCX input
    let optics = default_tissue_optics_830nm();
    let mcx_input = McxInput { /* ... populate all fields ... */ };

    // Write JSON
    let json_filename = format!("source_{}.json", source_index);
    let json = serde_json::to_string_pretty(&mcx_input)?;
    fs::write(out_dir.join(&json_filename), &json)?;

    Ok(McxExportResult {
        json_path: out_dir.join(&json_filename).to_string_lossy().to_string(),
        volume_path: out_dir.join(vol_filename).to_string_lossy().to_string(),
        source_voxel_pos: [source_voxel.x, source_voxel.y, source_voxel.z],
        volume_dim: dims,
    })
}
```

## Step 8: Load Fluence Results

After MCX runs, it produces fluence output files (`.jnii` or `.mc2`).

### 8.1 Parse .mc2 (simpler format)

The `.mc2` file is a flat array of `f32` values with dimensions Nx * Ny * Nz * Ng. For a single time gate (typical for CW-fNIRS), it is just Nx * Ny * Nz floats:

```rust
pub fn load_fluence_mc2(path: &Path, dims: [usize; 3]) -> Result<Array3<f32>, String> {
    let bytes = fs::read(path)?;
    let floats: Vec<f32> = bytes.chunks_exact(4)
        .map(|c| f32::from_le_bytes([c[0], c[1], c[2], c[3]]))
        .collect();
    // Reshape to 3D array
}
```

## Step 9: Jacobian Computation

This is the end goal. The Jacobian for a source-detector pair is computed from the fluence maps:

```
J(r) = fluence_source(r) * fluence_detector(r)
```

Where `r` is each voxel position. This gives you the sensitivity of each channel to absorption changes at each voxel. You then map this onto the cortex mesh vertices for the projection visualization.

This step connects back to Phase 4 of the main roadmap (Cortical Projection), replacing the simpler distance-based influence map with a physically accurate sensitivity profile.

## Critical Pitfall: Array Memory Layout

The single most common bug in MCX integrations is getting the volume byte order wrong. ndarray in Rust defaults to row-major (C order). MCX defaults to column-major (Fortran/MATLAB order, flag `-a 0`). You have two options:

Option A: Write row-major from Rust, pass `-a 1` (C order) to MCX
Option B: Transpose the array to column-major before writing, use MCX default

Choose one and document it clearly. Test by exporting a small asymmetric volume (e.g. 10x20x30 with a single non-zero voxel at a known corner) and verifying MCX sees it at the expected location.

## Critical Pitfall: Coordinate System Conventions

FreeSurfer uses RAS (Right-Anterior-Superior) coordinates. If your scene in Three.js uses a different convention (e.g. Y-up vs Z-up), you need an additional rotation in the transform chain. Document whatever convention your Viewport3D uses and account for it in `world_to_voxel`.
